# 개요

- 기존의 Git Flow 브랜치 전략은 현재의 웹/앱 방식을 고려하지 않았고 처음으로 제시했던 Vincent Driessen은 적합하지 않다고 회고하고 있다. 그 이유 중 하나는 branch의 생명주기가 너무 길어서 충돌이 일어날 확률이 크다는 것이다.
- 이에 대한 대안으로 가장 많이 사용되고 있는 방식 중 하나가 트렁크 기반 개발이다.

# 트렁크 기반 개발

- 기본적으로는 하나의 main 브랜치 위에서 운영되며, 모든 브랜치들이 main으로 직접 배포된다.
- 피처 브랜치를 보통 1-2일 정도만 열어놓는 규칙을 기반으로 더 작은 단위 배포가 가능해진다.
- 이러한 점 때문에 ㄷ테스트가 더 중요해진다.

단점

- 큰 기능을 개발할 때 브랜치 생명주기가 길어져서 규칙을 지키기 어려워진다.
- 피쳐 단위가 작아짐에 따라 앞에 피쳐가 개발되는 것의 의존성이 생긴다. (피쳐 토글을 활용해 해결해 볼 수 있다.)
- 바로 실서버에 배포되어 위험성이 존제한다. (이를 방지하기 위해 develop or staging을 하나 놓기도 한다.)

내 생각

- 확실한 PR(MR)에 대한 테스트, 리뷰, 정적분석, assigness가 존재한다면, git flow보다 더 좋은 방식일 수 있을 것 같다.
